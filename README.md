# LifeOS

Personal proactive AI accountability system. Full design lives in [00-MASTER_SPEC.md](00-MASTER_SPEC.md); quick context refresh in [01-HANDOFF.md](01-HANDOFF.md). This README covers day-to-day dev.

**Status:** Week 1 — foundation (Docker stack, DB schema, Telegram echo bot). See the spec's 6-week roadmap (§11) for what comes next.

## Quickstart

```bash
cp .env.example .env   # fill in TELEGRAM_BOT_TOKEN at minimum
docker compose up -d postgres redis
docker compose build app
docker compose run --rm app alembic upgrade head
docker compose up -d app n8n
curl localhost:8000/health
```

- FastAPI app: http://localhost:8000
- n8n: http://localhost:5678 (basic auth from `.env`)
- Postgres: localhost:5432

## Database migrations

Models live in `app/models.py`. After changing them, regenerate a migration:

```bash
docker compose run --rm app alembic revision --autogenerate -m "describe change"
docker compose run --rm app alembic upgrade head
```

Always review autogenerated migrations before applying.

## Telegram bot

1. Create a bot via [@BotFather](https://t.me/BotFather), put the token in `.env` as `TELEGRAM_BOT_TOKEN`.
2. Message [@userinfobot](https://t.me/userinfobot) to get your numeric Telegram user ID, put it in `.env` as `TELEGRAM_ALLOWED_USER_ID` (locks the bot to you only).
3. For real inbound delivery from Telegram's servers, the webhook needs a public HTTPS URL — see ngrok tunnel below. Until then you can exercise the handler locally:

```bash
curl -X POST localhost:8000/telegram/webhook \
  -H "Content-Type: application/json" \
  -d '{"message": {"from": {"id": YOUR_ID}, "chat": {"id": YOUR_ID}, "text": "hello"}}'
```

This should trigger a real outbound Telegram message back to you.

4. To use the driving-directions and train-departure tools, put a Google Maps API
   key (Directions API enabled) in `.env` as `GOOGLE_MAPS_API_KEY`. Share your
   location with the bot (attach → Location) before asking for driving
   directions — it's cached in Redis for 12 hours and used as the origin.

## Google Calendar (OAuth, one-time setup)

Not wired into the app yet (Week 2 work) — this just gets the credentials/refresh token ready.

1. In [Google Cloud Console](https://console.cloud.google.com), enable the Google Calendar API and create an OAuth client (Application type: **Desktop app**). Make sure the OAuth consent screen's **User type is External** (not Internal — Internal only works for Google Workspace org accounts) and add yourself as a test user.
2. Put `client_id`/`client_secret` in `.env` as `GOOGLE_CALENDAR_CLIENT_ID` / `GOOGLE_CALENDAR_CLIENT_SECRET`.
3. Run the one-time authorization script with the callback port published:
   ```bash
   docker compose run --rm -e PYTHONPATH=/code -e PYTHONUNBUFFERED=1 -p 8765:8765 app python -u scripts/google_calendar_auth.py
   ```
4. Open the printed URL in your own browser, approve access (click through the "Google hasn't verified this app" warning — expected for an app you own in Testing mode), and the script prints a refresh token. Put it in `.env` as `GOOGLE_CALENDAR_REFRESH_TOKEN`.

On Windows/Git Bash, prefix docker commands that use bare `/path` style arguments (like `-e PYTHONPATH=/code`) with `MSYS_NO_PATHCONV=1` — otherwise Git Bash silently rewrites `/code` into a Windows path.

## Web app

A multi-session chat UI (React SPA) living alongside the Telegram bot — see [03-WEBAPP-PLAN.md](03-WEBAPP-PLAN.md) for why and the full design. Same backend, same persona/tools, same single-tenant data; it's a second surface, not a second system. Telegram keeps its own always-on thread; the web app gives you unlimited disposable sessions plus Markdown-rendered replies.

**Login is Google OAuth, gated by an email allow-list** — not multi-tenancy. Every allowed email sees the same data; this only replaces the login mechanism. If you want a genuinely separate person (e.g. a family member with their own goals/expenses/calendar), that's still a separate deployment, same as the plan already on file for a second instance.

1. In the same [Google Cloud Console](https://console.cloud.google.com) project as your Calendar credentials, create a **second** OAuth client — Application type **Web application** this time (Calendar's is Desktop app and can't do a browser redirect flow). Add an authorized redirect URI of `https://YOUR_NGROK_STATIC_DOMAIN/api/auth/callback` (and `http://localhost:8000/api/auth/callback` too, if you want to test locally before the tunnel is up).
2. Put the client id/secret in `.env` as `GOOGLE_OAUTH_CLIENT_ID` / `GOOGLE_OAUTH_CLIENT_SECRET`, and the exact redirect URI you registered as `GOOGLE_OAUTH_REDIRECT_URI`.
3. Put the Google account email(s) allowed to log in as a comma-separated `WEBAPP_ALLOWED_EMAILS`.
4. Generate a random signing secret and put it in `.env` as `WEBAPP_SECRET_KEY`:
   ```bash
   openssl rand -hex 32
   ```
5. Build the frontend. `docker compose build app` does this automatically (the Dockerfile is a multi-stage build — see `webapp/` → `app/static/`), **but** since `docker-compose.yml` bind-mounts `./app` over the container's app directory for live code editing, that bind mount also shadows whatever the image baked in. So after any frontend change, rebuild the static output directly into `app/static/` (gitignored, generated) so the running container picks it up without a rebuild:
   ```bash
   cd webapp
   npm install   # first time only
   npm run build
   ```
   For frontend UI iteration without touching the backend, `npm run dev` inside `webapp/` runs Vite's dev server on :5173 with API calls proxied to :8000 (start `docker compose up -d app` separately first). The full Google OAuth round trip only works against a redirect URI you actually registered, though — test that against :8000 or the real tunnel, not the :5173 dev server.
6. Open `http://localhost:8000` (or your ngrok domain) and sign in.

## DB admin (Adminer)

Full read/write access to every table, for when you want to check or fix something directly instead of going through the assistant — see 03-WEBAPP-PLAN.md for why (the model has fabricated at least one fake write confirmation; don't trust chat replies alone for anything that matters).

Open `http://localhost:8081` and log in with:
- System: **PostgreSQL**
- Server: `postgres`
- Username / Password / Database: your `POSTGRES_USER` / `POSTGRES_PASSWORD` / `POSTGRES_DB` from `.env`

Deliberately **not** routed through the ngrok tunnel (check `docker-compose.yml` — the ngrok command only ever targets `app:8000`) since Adminer has no auth of its own and raw SQL edit access is a bigger risk than the app itself. Only reachable at `localhost:8081` on this machine or your home network; use Tailscale (or similar) if you want it while out, don't put it on the public tunnel.

## ngrok tunnel (manual setup, not automated)

The app is only reachable from the internet via an ngrok tunnel using a free reserved static domain — no port forwarding, no paid domain needed.

1. Sign up at [ngrok.com](https://ngrok.com) (free tier), copy your authtoken from the dashboard, put it in `.env` as `NGROK_AUTHTOKEN`.
2. Reserve a free static domain in the ngrok dashboard (Domains → New Domain), put it in `.env` as `NGROK_STATIC_DOMAIN` (e.g. `your-name.ngrok-free.dev`).
3. Start the tunnel service: `docker compose --profile tunnel up -d ngrok`.
4. Register the Telegram webhook against your static domain:
   ```bash
   curl "https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/setWebhook?url=https://YOUR_STATIC_DOMAIN/telegram/webhook"
   ```

Unlike Cloudflare's free Quick Tunnel, this URL is stable across restarts — no need to re-register the webhook each time. (If you later add your own domain to Cloudflare, a named Cloudflare Tunnel works the same way and is a drop-in alternative.)

## Secrets

`.env` is gitignored — never commit it. `.env.example` documents every variable with placeholders. Rotate the Telegram bot token via @BotFather (`/revoke`) and the OpenRouter key via its dashboard if either leaks.
